# SalonOS AI — פרומפט בנייה (MVP / שלב 1)

> פרומפט מוכן להעתקה לכלי בנייה כמו Claude Code, Cursor, Bolt או Lovable.
> ממוקד ב-MVP (שלב 1) בלבד — בניית כל 20 הסעיפים בבת אחת מובילה לבלגן.

---

## הפרומפט

```
אתה מהנדס תוכנה בכיר. אני בונה SaaS בשם "SalonOS AI" — מערכת פרימיום לניהול
מספרות, WhatsApp-first, בעברית מלאה (RTL), תואמת רגולציה ישראלית. תבנה איתי
את ה-MVP בצורה מודולרית, צעד אחר צעד. אל תקפוץ קדימה ואל תכתוב את כל הקוד בבת
אחת — תתחיל בארכיטקטורה ובסכמת הנתונים, ותעצור לאישור לפני כל מודול.

== מחסנית טכנולוגית ==
- Frontend: Next.js (React) + TypeScript, תמיכת RTL מהיסוד (logical CSS
  properties, פונט עברי, תאריכים/מספרים בעברית). UI נקי ומודרני.
- Backend: NestJS (Node) או FastAPI (Python) — בחר אחד והסבר למה.
- DB: PostgreSQL בארכיטקטורת multi-tenancy מאוחדת עם Row-Level Security
  (tenant_id מתוך ה-JWT, FORCE ROW LEVEL SECURITY, role לא-superuser).
- Redis לתורים והתראות (BullMQ/Celery).
- Auth: JWT + RBAC (בעל מספרה / מנהל / ספר / קבלה).

== טופולוגיה ולקוחות (Clients) ==
- Backend מתארח בענן (single source of truth). הוא מטפל ב-WhatsApp,
  מספרי הקצאה, AI, ובסנכרון בין כל המכשירים.
- מחשב המספרה (קבלה/מנהל): אפליקציית דסקטופ ב-Electron שעוטפת את ה-UI,
  עם גישה לחומרה מקומית (מדפסת קבלות, מגירת קופה, סורק ברקוד).
- עובדים: אפליקציית מובייל ב-React Native — צפייה ביומן, קביעת/שינוי תור,
  צפייה במשמרת האישית ובכרטיסי לקוח (לפי הרשאות RBAC).
- חובת Offline-first בלקוח הדסקטופ: בעת נפילת אינטרנט, היומן והקופה של
  היום ממשיכים לעבוד מ-cache מקומי; פעולות נכנסות לתור ומסתנכרנות אוטומטית
  כשהחיבור חוזר (sync queue + פתרון התנגשויות last-write-wins עם חותמת זמן).
- כל המכשירים רואים את אותו יומן בזמן אמת (WebSockets) — תור שנקבע בטלפון
  של ספר מופיע מיד על מחשב הקבלה.

== עקרונות חובה (לא מתפשרים) ==
1. בידוד דייר ברמת ה-DB — אסור שאילתה תוכל לדלוף נתונים בין מספרות.
2. כל שורת UI ב-RTL ובעברית. ערבוב עברית/אנגלית/מספרים מטופל נכון (bidi).
3. תשתית הסכמה (consent ledger) ופונקציית opt-out מובנות מהיום הראשון
   (חוק הספאם / סעיף 30א — אסור לשלוח הודעת שיווק בלי הסכמה מתועדת).
4. תשתית מוכנה לחשבונית ישראל (בקשת "מספר הקצאה" אוטומטית) ולספקי סליקה
   ישראליים — הגדר interface מופשט, אל תקבע ספק קשיח.
5. Audit logging על כל פעולה רגישה. הצפנה at-rest ו-in-transit. תמונות לקוח
   נחשבות מידע רגיש (תיקון 13) — אחסון מבודד + הסכמה מפורשת.

== היקף ה-MVP (שלב 1 בלבד) ==
א. לוח בקרה למנהל — הכנסות יום/חודש מול יעד, תורים להיום, אחוז תפוסה,
   שעות פנויות.
ב. יומן ותורים — multi-stylist, drag-drop, בחירת שירות/ספר/משך/מחיר,
   רשימת המתנה, אישור הגעה.
ג. כרטיס לקוח — פרטים, היסטוריית טיפולים, נוסחאות צבע, תמונות לפני/אחרי
   (עם הסכמה), הערות.
ד. WhatsApp — אישורי תור ותזכורות (הודעות utility) דרך BSP רשמי
   (360dialog/Twilio). נצל את חלון השירות של 24 שעות כדי לחסוך עלויות.
   הגדר interface ל-BSP, אל תקבע ספק קשיח.
ה. POS וחיוב — סליקת אשראי + הפקת חשבונית ירוקה עם מספר הקצאה (דרך
   Morning/iCount). תמיכה בתשלומים/Bit.
ו. מניעת ביטולים בסיסית — תזכורות + דרישת מקדמה לטיפולים יקרים/בסיכון
   (כללים, לא ML בשלב הזה).
ז. ניהול עובדים בסיסי + דוחות בסיסיים.

== מה לא לבנות עכשיו (שלבים 2–3, אל תיגע) ==
בוט AI שיחתי, מודל ML לחיזוי ביטולים/נטישה, הדמיית שיער, תמחור דינמי,
מנויים, מועדון VIP, שיווק אוטומטי, "יועץ עסקי AI", ריבוי סניפים. תכנן
שהקוד יתרחב לשם בעתיד, אבל אל תממש כלום מזה כרגע.

== מה אני רוצה ממך עכשיו ==
1. תרשים ארכיטקטורה ותיקיות הפרויקט.
2. סכמת ה-DB המלאה (טבלאות + מדיניות RLS) כקובץ SQL.
3. רשימת ה-modules לפי סדר בנייה.
תעצור אחרי זה לאישור שלי לפני שאתה כותב קוד אפליקציה.
```

## הערות חשובות לפני הרצה

- **הסכמה והחשבונית הן תשתית, לא פיצ'ר.** לכן הן בפרומפט מההתחלה. אם תשאיר אותן
  לסוף — תצטרך לשכתב חלקים גדולים.
- **הפרומפט מבקש לעצור לאישור** אחרי הארכיטקטורה. קריטי: בלי זה כלי AI מייצרים
  אלפי שורות קוד שבורות שקשה לתקן. עדיף מודול-מודול.
- **Bolt / Lovable** מצוינים לאב-טיפוס UI מהיר, אבל פחות טובים ב-multi-tenancy עם
  RLS ובאינטגרציות ישראליות. את שכבת ה-backend והרגולציה עדיף לבנות ב-Claude Code
  או Cursor על קוד אמיתי.
